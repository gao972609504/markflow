功能规格说明：光模块管理

**功能分支**：`feature-xcvr-mgmt-integrated` **创建时间**：2026-06-09 **状态**：草案 **输入**：整合 sfp-mgmt 目录下所有需求

## 架构设计背景

### 早期架构（已废弃）

**硬件拓扑**：`CPU --- I2C --- PORT CPLD --- SFP present、lpmode等信号`

**工作模式**：CPU 使用周期性查询模式来判断光模块插入事件和拔出事件

**存在问题**：

- 查询模式实时性差，无法保证拔插事件的及时检测
- 光模块数量增多时，查询周期与系统开销之间的矛盾日益突出
- 可能丢失或误报拔插事件，影响系统可靠性

### 当前架构

**硬件拓扑**：`CPU --- PCIE --- FPGA --- 私有串行总线 --- PORT CPLD --- SFP present、lpmode等信号`

**工作模式**：拔插事件发生时，FPGA 上送 MSI-X 中断给 CPU，由周期性查询变为中断通知

**架构优势**：
- **实时性**：中断通知模式可快速响应拔插事件（延迟 < 5ms）
- **准确性**：硬件事件锁存机制保证不丢失最终状态
- **可扩展性**：支持 128 端口，可扩展至 256 端口

**当前实现**：
- 拔插事件：通过 FPGA 中断获取（FPGA 寄存器缓存事件记录）
- 单端口在位查询：直接读取 PORT CPLD 原始信息

### 后续优化方向

**单端口查询优化**：查询单端口在位状态从读取 PORT CPLD 改为读取 FPGA 寄存器

**优化收益**：

- 统一数据来源，提高一致性
- 充分利用 FPGA 寄存器缓存能力，提升性能

## 用户场景与测试 *(必填项)*

### Driver 模块用户故事

#### 用户故事 1.1 - 单端口在位状态查询（优先级：P1）

UXOS xcvr 模块使用 new platform API sfp 模块中的 get_present 接口来获取光模块在位状态，这是光模块管理的基础能力，所有后续操作（如 I2C 访问、链路 up/down）都依赖于此信息。

同时，运维人员也需要实时查询特定端口的光模块在位状态，以便进行端口级别的故障诊断和配置管理。例如，在排查端口链路故障时，首先需要确认光模块是否在位。

**优先级说明**：这是光模块管理的核心基础功能，UXOS xcvr 模块和运维管理都依赖于此接口，因此设为 P1 最高优先级。

**独立测试方式**：可通过逐个插入/拔除光模块并读取对应端口的在位状态节点来验证。

**验收场景**：

1. **前提** UXOS xcvr 模块初始化，**操作** 调用 get_present 接口，**结果** 正确返回光模块在位状态
2. **前提** 光模块插入，**操作** UXOS 通过 get_present 查询，**结果** 返回"1"或"present"，触发后续初始化流程
3. **前提** 光模块拔出，**操作** UXOS 通过 get_present 查询，**结果** 返回"0"或"absent"，触发清理流程
4. **前提** 端口无光模块，**操作** 插入光模块，**结果** 对应端口的在位状态节点应立即显示"1"或"present"
5. **前提** 端口有光模块，**操作** 拔除光模块，**结果** 对应端口的在位状态节点应立即显示"0"或"absent"
6. **前提** 读取端口在位状态，**操作** 连续读取同一端口多次，**结果** 每次读取应返回一致的状态

**sysfs 节点示例**：

```bash
# 单端口在位状态查询节点
/sys_switch/transceiver/eth1/present

# 节点返回值
cat /sys_switch/transceiver/eth1/present
1   # 1=在位，0=不在位

# 节点实现（软链接指向 PORT CPLD 原始设备节点）
ls -l /sys_switch/transceiver/eth1/present
lrwxrwxrwx 1 root root 41 Jun  9 11:15 present -> /sys/bus/i2c/devices/47-0053/xcvr_present
```

#### 用户故事 1.2 - 批量拔插事件检测（优先级：P1）

UXOS xcvr 模块使用 new platform API sfp 模块中的 get_change_event 接口来获取光模块插入和拔出事件，用于处理光模块的初始化和管理。这是光模块热插拔事件的核心通知机制，系统需要根据这些事件来触发模块初始化、资源分配、链路配置等操作。

驱动需要提供统一接口一次性获取所有端口的拔插事件记录，支持上层应用快速了解端口变化情况。

**优先级说明**：这是光模块热插拔管理的核心功能，UXOS xcvr 模块依赖此接口来处理光模块的初始化和管理，是 P1 最高优先级。

**独立测试方式**：多次拔插光模块后读取事件记录节点，验证最终状态与硬件实际状态一致。

**验收场景**：

1. **前提** UXOS xcvr 模块初始化，**操作** 调用 get_change_event 接口，**结果** 正确返回所有端口的拔插事件状态
2. **前提** 光模块插入，**操作** UXOS 通过 get_change_event 查询，**结果** 检测到插入事件，触发光模块初始化流程
3. **前提** 光模块拔出，**操作** UXOS 通过 get_change_event 查询，**结果** 检测到拔出事件，触发资源清理流程
4. **前提** 系统初始状态无拔插事件，**操作** 在端口1和端口3先后插入光模块，**结果** 读取事件节点显示端口1和端口3的插入事件，最终状态为在位
5. **前提** 多个端口同时拔插，**操作** 一次性读取事件记录节点，**结果** 所有端口的拔插事件按时间顺序完整记录，无遗漏
6. **前提** 系统正常运行，**操作** cat /sys_switch/transceiver/sff_change_event_all，**结果** 返回128字符的字符串，格式：0=未变化，1=插入，2=拔出

#### 用户故事 1.3 - 光模块上下电控制（优先级：P1）

驱动需要提供整机光模块电源控制功能，通过 CPLD 寄存器实现电源管理。

**优先级说明**：电源控制是节能和故障隔离的基础功能，是 P1 级别需求。

**独立测试方式**：通过 sysfs 节点控制整机光模块电源，验证控制效果。

**验收场景**：

1. **前提** 系统正常运行，**操作** 写入0到 /sys_switch/transceiver/power_on，**结果** 整机光模块下电
2. **前提** 光模块已下电，**操作** 写入1到 /sys_switch/transceiver/power_on，**结果** 整机光模块上电
3. **前提** 读取电源状态，**操作** cat /sys_switch/transceiver/power_on，**结果** 返回0（未上电）或1（已上电）
4. **前提** 光模块下电状态，**操作** 读取光模块在位状态 `/sys_switch/transceiver/eth1/present`，**结果** 在位检测功能正常工作，返回正确的在位信息

**功能独立性说明**：

光模块上下电控制与在位检测功能是独立的：
- 光模块下电后，光模块的在位检测功能正常工作
- 光模块在位检测、电源管理、reset 管理功能是独立且分开的功能

**设计原则**：

CPLD 层面只提供基础控制能力，不做业务逻辑组合：
- CPLD 实现了光模块的独立下电功能
- CPLD 并没有做逻辑完整功能的设计
- Driver 独立抛出在位检测、电源管理、reset 管理节点和功能
- 组合业务逻辑由 UXOS 或者客户 NOS 来实现

**独立测试方式**：模拟 reboot、modprobe/rmmod、上下电后验证状态同步；触发 CPLD/FPGA reset 验证检测和恢复机制。

**验收场景**：

1. **前提** 系统正常运行部分端口有光模块，**操作** 执行系统 reboot，**结果** 重启完成后所有端口在位状态与实际硬件状态一致
2. **前提** 驱动已加载，**操作** 执行 modprobe -r 卸载后重新加载，**结果** 驱动重新加载后重新初始化所有状态，同步在 500ms 内完成

4. **前提** 系统正常运行，**操作** 触发 FPGA reset，**结果** 驱动在 100ms 内检测到 reset 事件并重新初始化 FPGA 寄存器接口

#### 用户故事 1.6 - DFT 测试需求：模拟光模块在位/不在位 功能（优先级：P1）

UXOS 需要一个自动化测试方案验证光模块管理功能的稳定性。光模块拔插事件检测涉及完整的链路：HW → PORT CPLD → FPGA → CPU → Driver → UXOS PMON → SAI → 最终用户看到的端口 Link UP。仅靠人工拔插无法满足测试需求，且耗费人力。

因此需要一个 simulation 模块在位/不在位的方法，用于模拟光模块拔插事件。

**优先级说明**：光模块管理是核心功能，需要稳定性验证。自动化测试是保证核心功能可靠性的关键，人工拔插无法满足回归测试和压力测试需求，因此调整为 P1 级别。

**独立测试方式**：通过 DFT 模拟接口执行自动化测试序列，验证完整链路的拔插事件处理和端口 Link UP 流程。

**验收场景**：

1. **前提** 测试模式开启，**操作** 设置光模块在位模拟值为不在位，**结果** Driver 获取到不在位信息，与实际光模块在位情况无关
2. **前提** 模拟不在位状态，**操作** 设置光模块在位模拟值为在位，**结果** Driver 获取到在位信息，触发完整的中断上报链路
3. **前提** 模拟在位状态，**操作** 验证端口 Link UP，**结果** 端口正常 Link UP，完整链路验证通过
4. **前提** 测试模式关闭，**操作** 读取光模块在位信息，**结果** 返回光模块真实的在位情况

**详细需求说明**：

**在位信息传递机制**：
- 光模块在位信息传递给 FPGA 模块
- PORT CPLD 的在位寄存器（CPLD Reg）仅用于 debug，Driver 默认不导出 sysfs 节点
- 默认情况下获取的在位信息为光模块真实的在位情况，不跟光模块上下电产生关联

**测试模式需求**：
- 需要一个测试模式开关
- 测试模式开启时，获取光模块在位信息根据设置的模拟值返回，跟实际的光模块在位情况无关
- 测试模式关闭时，返回光模块真实的在位情况

**设计建议（基于 PORT CPLD 进行最接近物理拔插的模拟）**：

**寄存器定义：CON_SIMULATE_PRESENCE**

| Bit | Symbol | Access | Description | Reset value |
|-----|--------|--------|-------------|-------------|
| 1:0 | - | - | Reserved | - |
| 2 | EN | R/W | Simulate the presence of the optical module when set to 1 | 0 |
| 7:3 | - | - | Reserved | - |

开启模拟光模块在位的功能，开启后可以设置 CON_OPTICAL_IO2 的 bit 6 Set_ModPrsL 实现光模块在位信号的模拟。

**寄存器定义：CON_OPTICAL_IO2**

| Bit | Symbol | Access | Description | Reset value |
|-----|--------|--------|-------------|-------------|
| 1:0 | - | - | Reserved | - |
| 3 | ePPS/Clock | R/W | - | 0 |
| 4 | P/VS1 | R/W | - | 0 |
| 5 | P/VS4 | R/W | - | 0 |
| 6 | Set_ModPrsL | R/W | Set the optical module presence information in debug mode. Setting it to 1 indicates that the module is present, and setting it to 0 indicates that it is not present. | 0 |
| 7 | - | - | Reserved | 0 |

**使用场景：测试模式下模拟光模块拔插动作（先拔出，然后插入）**

具体步骤：
1. 打开光模块在位模拟模式
   - set CON_SIMULATE_PRESENCE.2 = 1
2. 设置光模块在位模拟值为不在位
   - set CON_OPTICAL_IO2.6 = 0
3. 将写保护关闭
   - set CON_WP.2 = 0
4. 设置光模块下电
   - set CON_POWER.2 = 0
5. 设置光模块在位模拟值为在位
   - set CON_OPTICAL_IO2.6 = 1
6. 设置光模块上电
   - set CON_POWER.2 = 1
7. 取消光模块在位模拟模式
   - set CON_SIMULATE_PRESENCE.2 = 0
8. 将写保护打开
   - set CON_WP.2 = 1

**测试覆盖范围**：
完整的联调稳定性测试链路：HW → PORT CPLD → FPGA → CPU → Driver → UXOS PMON → SAI → 最终用户看到的端口 Link UP

#### 用户故事 1.7 - 光模块 Reset 管理（优先级：P1）

针对支持 reset 功能的光模块，PORT CPLD 需要在整机 reboot 和整机 power cycle 时自动 reset 光模块，确保光模块处于确定的初始状态。

**优先级说明**：光模块 reset 是确保光模块在系统重启后处于确定初始状态的关键功能，避免光模块保持上一次的异常状态，是 P1 级别需求。

**独立测试方式**：执行系统 reboot 和整机 power cycle，验证光模块 reset 行为。

**验收场景**：

1. **前提** 系统正常运行，**操作** 执行系统 reboot，**结果** PORT CPLD 在重启过程中 reset 所有支持 reset 功能的光模块
2. **前提** 系统正常运行，**操作** 执行整机 power cycle（断电后重新上电），**结果** PORT CPLD 在上电过程中 reset 所有支持 reset 功能的光模块
3. **前提** 光模块完成 reset，**操作** 检查光模块状态，**结果** 光模块处于确定的初始状态，可以进行正常初始化

**功能说明**：

**Reset 触发条件**：
- 整机 reboot（软件重启）
- 整机 power cycle（断电后重新上电）

**Reset 对象**：
- 仅针对支持 reset 功能的光模块
- 不支持 reset 功能的光模块不受影响

**Reset 目的**：
- 确保光模块在系统重启后处于确定的初始状态
- 避免光模块保持上一次的异常状态
- 为光模块的正常初始化提供干净的起始状态

**功能独立性说明**：

光模块 reset 管理与在位检测、电源管理功能是独立的：
- 光模块 reset 后，在位检测功能正常工作
- 光模块在位检测、电源管理、reset 管理功能是独立且分开的功能
- CPLD 层面只提供基础 reset 能力，不做业务逻辑组合
- Driver 独立抛出 reset 管理节点和功能（如有需要）

#### 用户故事 1.8 - 在位事件延时上报（优先级：P1）

检测到光模块在位后，PORT CPLD 需要延时一个 delay 时间后再给 CPU 上送在位事件，确保光模块稳定后再通知系统。

**优先级说明**：在位事件延时上报是避免光模块未完全稳定时系统过早访问的关键机制，可防止通信错误，是 P1 级别需求。

**独立测试方式**：插入光模块后测量从物理插入到 CPU 收到中断的时间，验证延时机制。

**验收场景**：

1. **前提** 光模块插入，**操作** 测量延时时间，**结果** PORT CPLD 延时默认 200ms 后上送在位事件
2. **前提** 光模块插入，**操作** 验证系统收到中断时的时间，**结果** 光模块已稳定，系统可以正常访问光模块
3. **前提** 光模块快速插拔，**操作** 验证事件上报，**结果** 延时机制有效过滤不稳定状态

**功能说明**：

**延时机制**：
- 检测到光模块在位信号后，启动延时计时
- 延时期间，在位信号保持稳定则上送事件
- 延时期间，在位信号变化则重新计时

**默认延时时间**：
- 默认延时：200ms
- 延时时间可根据实际光模块特性调整

**延时目的**：
- 确保光模块完全插入并稳定
- 避免光模块未完全稳定时系统过早访问
- 防止通信错误和访问失败
- 提高系统稳定性

**与防抖的关系**：
- CPLD 硬件防抖：过滤微秒~毫秒级的机械触点抖动
- 延时上报：确保光模块电路稳定和 I2C 通信就绪
- 两者配合使用，提供完整的稳定性保障

### DiagOS 模块用户故事

#### 用户故事 2.1 - LPMode 默认关闭配置（优先级：P1）

DiagOS 系统需要默认关闭 LPMode 自动管理功能，光模块在上电和插入后不自动进入低功耗模式。

**优先级说明**：DiagOS 是诊断系统，需要光模块立即可用，默认关闭 LPMode 是 P1 级别配置需求。

**独立测试方式**：在 DiagOS 系统上检查 LPMode 默认配置。

**验收场景**：

1. **前提** DiagOS 系统启动，**操作** 检查 LPMode 配置，**结果** 默认关闭自动管理
2. **前提** DiagOS 下插入光模块，**操作** 检查 LPMode 状态，**结果** 光模块不自动进入 LPMode
3. **前提** DiagOS 下插入光模块，**操作** 检查 link 状态，**结果** 端口可以正常 link UP

**控制接口说明**：

**节点路径**：`/sys_switch/transceiver/eth[n]/auto_lpmode_ctrl`

**读写格式**：String
- 写入 "1"：使能 CPLD 自动管理功能
- 写入 "0"：去使能 CPLD 自动管理功能

**功能机制与行为详解**：

此功能的核心是 CPLD 内置的硬件状态机，其设计用于在特定硬件事件发生时，自动管理光模块的功耗状态，以确保系统稳定性和模块安全。

**自动管理触发条件**：

当检测到以下任一硬件事件时，已使能的 CPLD 逻辑会自动将光模块的 lpmode 引脚置为有效状态（即低功耗模式）：
- 系统首次上电 (Power-On)
- 光模块被拔出 (Module Removal)
- 光模块被插入 (Module Insertion)

（其本质是监测光模块的"在位信号"(present) 的电平变化）

**功能影响与软件交互**：

- **进入低功耗**：一旦 CPLD 自动将模块置入低功耗模式，光模块的数据通道将被关闭，无法进行正常业务通信。
- **退出低功耗**：必须由设备软件（驱动或管理进程）通过写 lpmode 控制寄存器等显式操作，使模块退出低功耗模式，恢复正常工作。

**默认状态与配置策略**：

- **默认策略（上电使能）**：系统上电后，该功能默认处于使能状态 (auto_lpmode_ctrl=1)。这是一种保守的安全设计，确保任何非预期状态（如上电瞬态、热插拔电弧）都不会使模块处于不受控的工作状态。
- **软件干预流程**：系统软件在启动或检测到模块插入后，需执行完整的初始化序列，其中关键一步即为手动清除模块的低功耗状态。
- **关闭自动管理**：若软件写入 0 以去使能此功能，则 CPLD 将不再响应上述硬件事件。此后，lpmode 引脚的状态将完全由软件或其他静态配置决定，CPLD 不再进行自动干预。

**功能说明**：
- 用于使能或去使能 CPLD 对光模块低功耗模式(lpmode)的自动管理功能
- BTS 新增接口
- DiagOS 默认配置为 "0"（去使能）

**使用示例**：
```bash
# DiagOS 默认关闭 LPMode 自动管理
cat /sys_switch/transceiver/eth1/auto_lpmode_ctrl
0   # 0=去使能，1=使能

# 使能 LPMode 自动管理
echo "1" > /sys_switch/transceiver/eth1/auto_lpmode_ctrl

# 去使能 LPMode 自动管理
echo "0" > /sys_switch/transceiver/eth1/auto_lpmode_ctrl
```

## 边缘场景

### 通信与数据完整性场景
- 当 CPLD-FPGA 总线通信完全中断时，驱动如何报告错误状态？驱动应通过 sysfs 节点报告总线通信失败，并保持最后一次有效状态或标记为未知
- 当 FPGA 寄存器读取返回无效数据时，系统如何处理？驱动应检测数据有效性（如校验和、超时等），并返回错误代码而非无效数据
- 当光模块在驱动初始化过程中拔插，事件是否会丢失？驱动应确保在初始化完成后读取 FPGA 缓存的事件记录，防止遗漏

### 正常场景边缘情况（上下电、Reboot、驱动加卸载）
- 当系统在驱动未完全初始化时发生 reboot，如何保证状态一致性？驱动应在初始化时主动查询硬件状态，而非依赖历史记录
- 当驱动卸载期间光模块状态发生变化，重新加载后如何处理？驱动应读取当前硬件状态作为基准
- 当上下电过程中 CPLD 和 FPGA 上电时序不一致时，如何处理？驱动应在 FPGA 就绪后延迟足够时间（如 100ms）再读取 CPLD 状态

### 运行时异常场景（CPLD/FPGA Reset）
- 当 CPLD 发生 reset 但驱动未及时检测到，会导致什么问题？驱动应定期轮询 CPLD 状态寄存器（如每 50ms），确保检测延迟不超过 100ms
- 当 FPGA 发生 reset 导致寄存器内容清空，事件记录如何恢复？驱动应检测 reset 事件并清除内存中的事件缓存，重新开始记录
- 当多次连续 reset 发生时，驱动如何记录？应记录每次 reset 的时间戳和类型，并保留最近至少 16 次的 reset 历史记录

### DFT 模拟相关
- 当 DFT 模拟注入和真实物理事件同时发生时，系统按 DFT优先级处理
- 当 DFT 模拟功能未使能时，DFT 注入寄存器写操作不应影响正常运行状态

## 需求 *(必填项)*

### 功能需求

**Driver 基础需求**：
- **FR-001**：系统必须支持对 128 个端口的光模块进行实时在位检测，通过查询接口读取每个端口的真实物理在位状态
- **FR-002**：系统必须在光模块发生插入或拔出事件时，通过 MSI-X 中断实时通知 CPU
- **FR-003**：每个端口必须提供独立的 sysfs 节点（`/sys_switch/transceiver/eth[N]/present`），读取时返回该端口当前在位状态（present/absent）
- **FR-004**：系统必须提供 /sys_switch/transceiver/sff_change_event_all 节点，用于一次性获取所有端口模块的拔插 event
- **FR-005**：系统必须提供整机光模块电源控制功能，提供 /sys_switch/transceiver/power_on 节点
- **FR-006**：节点写入0表示光模块未上电，写入1表示光模块已上电

**状态恢复与异常处理需求**：
- **FR-007**：驱动在系统 reboot 或上下电后重新加载时，必须主动扫描所有端口并同步当前硬件实际状态
- **FR-008**：驱动必须支持 modprobe 卸载后重新加载，重新加载后应重新初始化所有状态并与硬件同步

**调试与测试需求**：
- **FR-009**：驱动必须提供 PORT CPLD 原始 GPIO PIN 状态的 sysfs 节点
- **FR-010**：驱动必须记录 CPLD-FPGA 内部并行总线的错误情况，并通过 sysfs 节点暴露
- **FR-011**：驱动必须提供完整帧缓存的 sysfs 节点，保留最近 64 帧完整数据
- **FR-012**：系统必须提供 DFT 模拟注入接口，允许软件写寄存器来模拟任意端口的插入或拔出事件
- **FR-013**：系统必须提供 DFT 使能控制位，只有当 DFT 使能时，模拟注入才生效

**Reset 管理需求**：
- **FR-014**：PORT CPLD 必须在整机 reboot 时 reset 所有支持 reset 功能的光模块
- **FR-015**：PORT CPLD 必须在整机 power cycle 时 reset 所有支持 reset 功能的光模块
- **FR-016**：PORT CPLD reset 仅针对支持 reset 功能的光模块，不影响不支持 reset 的光模块
- **FR-017**：驱动必须能够检测 CPLD reset 事件，并在检测到 reset 后重新扫描所有端口状态
- **FR-018**：驱动必须能够检测 FPGA reset 事件，并在检测到 reset 后重新初始化 FPGA 寄存器接口

**在位事件延时上报需求**：
- **FR-019**：PORT CPLD 检测到光模块在位后必须延时上报，默认延时 200ms
- **FR-020**：延时期间在位信号变化时必须重新计时

**LPMode 控制需求**：
- **FR-021**：系统必须提供 auto_lpmode_ctrl 节点控制 CPLD 的 LPMode 自动管理功能
- **FR-022**：写入1使能 CPLD 自动管理，写入0去使能 CPLD 自动管理
- **FR-023**：系统必须提供 low_power_mode 节点用于手动控制光模块进入或退出 LPMode
- **FR-024**：UXOS 默认开启 LPMode 自动管理
- **FR-025**：百度客户默认开启 LPMode 自动管理
- **FR-026**：DiagOS 默认关闭 LPMode 自动管理
- **FR-027**：CFI 客户默认关闭 LPMode 自动管理

**产线检测需求**：
- **FR-028**：系统必须提供产线焊接质量检测功能，检测 PORT CPLD 到 FPGA 之间的私有串行总线通信质量
- **FR-029**：焊接质量检测必须通过 CPLD-FPGA 总线读写测试完成
- **FR-030**：检测到通信错误时，错误记录必须包含错误类型、发生时间和相关端口信息
- **FR-031**：焊接质量检测程序必须在 10 秒内完成

**性能需求**：
- **FR-032**：系统必须准确上报光模块插入和拔出事件，不能丢失最终事件

### 核心实体

- **光模块端口（Xcvr Port）**：代表一个可插拔光模块的物理端口，包含端口索引、在位状态、拔插事件历史
- **拔插事件（Plug Event）**：代表一次光模块的插入或拔除动作，包含端口索引、事件类型（插入/拔除）、事件时间戳
- **PORT CPLD 状态（CPLD Status）**：代表 PORT CPLD 的原始硬件状态，包含 GPIO PIN 状态数组、总线错误计数
- **LPMode**：Low Power Mode，低功耗模式
- **LPMode Control Register**：LPMode 控制寄存器，控制光模块功耗状态的寄存器
- **DFT 模拟值（DFT Simulated Value）**：128 位寄存器，每端口 1 bit，用于模拟光模块在位状态

## 假设条件

**基础架构假设**：
- PORT CPLD 和 FPGA 已经完成硬件设计，寄存器接口定义明确
- PORT CPLD 通过内部并行总线将 GPIO 状态传递给 FPGA
- FPGA 提供统一的后端寄存器接口，CPU 通过配置空间（如 PCIe）或 memory mapped IO 访问
- 驱动运行在 Linux 操作系统上，遵循标准的 sysfs 接口规范
- 最大支持 128 个端口，可扩展到 256 端口

**LPMode 相关假设**：
- CPLD 支持 LPMode 自动管理功能
- 光模块支持 LPMode
- 各个 OS 能够正确配置 LPMode

**产线检测相关假设**：
- PORT CPLD 和 FPGA 之间的私有串行总线支持读写测试
- 焊接质量问题可以通过通信错误检测来发现
